- Sidecar frontmatter-consistency bug in `write_compaction_sidecar.py`: the recorded `last_sync_hashes` and the embedded `baseline_writable_content` can disagree — i.e., `sha256(baseline_writable_content['<section>']) != last_sync_hashes['<section>']`. Observed in this session for `# Follow-up Ideas`:
    - `last_sync_hashes['# Follow-up Ideas'] = a964466747626c6254d...`
    - `sha256(baseline_writable_content['# Follow-up Ideas']) = e04e8cb9a70cab14d79...`
  When this skew exists, `--mode verify` can never pass even after `--mode recover-main-writables` — recover restores from `baseline_writable_content`, but the verify check uses `last_sync_hashes`. There is no in-band recovery path; the sidecar must be regenerated from a clean main state. Suspected cause: a mirror mechanism (or another concurrent write) modifies main's writable section between the two hash captures during `--mode create`. Durable fix candidates:
    - (a) Compute `last_sync_hashes` from `baseline_writable_content` (the embedded snapshot) rather than re-reading main — guarantees consistency by construction;
    - (b) Add a self-consistency assertion at the end of `--mode create` that recomputes hashes from `baseline_writable_content` and fails create if they diverge from `last_sync_hashes` (fail-fast rather than later failure at sync-back);
    - (c) Document the failure mode + recovery (regenerate sidecar) in the sidecar SKILL's halt-trigger-#6 protocol so future agents recognize this case quickly.
  This is related to (and likely shares root cause with) the recurring-sidecar-dirty-worktree pattern captured earlier this session. Should be promoted to a permanent idea file via `/capture-idea` at wrap-up.
