## Session-discovered tooling friction

### setup_pwi_worktree.py + sidecar placement for multi-repo plans

**Symptom**: For a plan that lives in agentic-workspace (`library/docs/planning/projects/<workspace>/...`) but modifies code in another repo (e.g., TRITON-SWMM_toolkit), `setup_pwi_worktree.py` auto-detects target workspace from the plan's host repo (agentic-workspace) and creates a worktree there. The agentic-workspace worktree is then near-empty because planning docs live on main and are sparse-checkout-excluded. The code-implementation worktree gets created separately by Claude Code's `EnterWorktree` based on session CWD. Result: two worktrees with the same slug across two repos and no automatic alignment between them.

**Concrete failure surface**: `write_compaction_sidecar.py --worktree {path}` takes whatever path the caller passes — it does not enforce that the worktree is the one holding the code about to be modified. When the caller defaults to the `setup_pwi_worktree`-created worktree (which is the wrong one for project plans), the sidecar lands in the agentic-workspace worktree's `sidecars/` directory. A post-clear resume agent CWD'd in the toolkit worktree must know to read the sidecar from the OTHER worktree's path — an inversion that breaks the sidecar locality invariant. Caught in session 2026-05-15; sidecar moved manually to the toolkit worktree.

**Additional friction observed**: the toolkit worktree's `sidecars/` directory contained legacy sidecars from prior 2026-05-14 sessions despite the worktree being created fresh on 2026-05-15. `EnterWorktree` appears to preserve the `.claude/worktrees/<slug>/` subdirectory contents across worktree-create cycles (the `<slug>` directory exists across multiple sessions because it's named deterministically from the scratch doc stem). `--mode create` rejects with 'sidecars already exist' even though those sidecars are stale prior-session artifacts. Workaround: manually move stale sidecars to a `_legacy_<date>/` subdir before retry.

**Proposed fix directions** (capture for triage, not commit):
1. `setup_pwi_worktree.py` should infer project workspace from the plan's path (`projects/<workspace>/` substring) OR from `workspace:` frontmatter on any phase doc when absent on the master, and create the worktree in the project repo, not the agentic-workspace repo.
2. `write_compaction_sidecar.py` should require the worktree path's git remote to match (or be compatible with) the plan's code-affecting repo, OR emit a warning when the two repos don't agree.
3. `EnterWorktree` (Claude Code harness behavior) preserving stale `sidecars/` directory contents across fresh worktree creates is either a feature (resume continuity across sessions) or a bug — clarify and document either way. If feature: add a `--mode create-overwrite` or auto-archive on stale-detection.

**Related**: the master plan's frontmatter does NOT declare `workspace: TRITON-SWMM_toolkit` (only phase docs do). This made auto-detection harder. Consider adding a planning-frontmatter audit rule that requires master plans to declare `workspace:` when phase children declare it.
