# Implementing SLURM Cancellation Function

## Summary

This plan implements `analysis.cancel()` for batch job workflows with these key features:

1. **Zero-conditional API**: Just call `analysis.cancel()` - no need to check if jobs are running first
2. **Verified cancellation**: Waits for jobs to actually terminate, not just sends signals
3. **Cross-session persistence**: Close terminal, reopen, call `cancel()` - works seamlessly
4. **Informative messaging**: Clear output whether jobs were canceled or already completed
5. **Snakemake job naming**: Leverages `analysis_id` in job names for easy identification

**Simple usage**:
```python
tk = Toolkit("system.yaml", "analysis.yaml")
tk.analysis.cancel()  # That's it - handles everything
```

## Context

Users need the ability to cancel ongoing `batch_job` workflows programmatically via `analysis.cancel()`. This is critical for HPC workflows where:

1. **Long-running orchestrators**: The Snakemake orchestrator job can run for hours/days
2. **Many worker jobs**: Each simulation spawns separate SLURM jobs via Snakemake's SLURM executor
3. **Terminal sessions end**: Users close terminals and need to cancel from a fresh session
4. **Multiple analyses running**: Need to identify and cancel specific workflows without affecting others

Currently, users must manually:
- Identify the orchestrator job ID by inspecting `squeue` output
- Find all worker jobs (often with cryptic names)
- Run `scancel` commands for each job type

This plan implements persistent workflow tracking with Snakemake job name customization to enable simple programmatic cancellation.

## Key Design Decisions

### 1. Use Analysis Log for Persistence (Not Separate Registry)

**Decision**: Store orchestrator metadata in `TRITONSWMM_analysis_log` using the existing `LogField[T]` pattern.

**Rationale**:
- Analysis log already persists to `{analysis_dir}/log.json`
- Automatically loaded when analysis is reconstructed across sessions
- Follows established patterns (e.g., `all_sims_run`, `all_scenarios_created`)
- No extra files or infrastructure needed
- Type-safe with Pydantic validation

**Alternative rejected**: Creating a separate `~/.triton_swmm/workflows` registry
- Would require separate file management
- More complex cross-referencing between analysis and registry
- Duplication of analysis_dir path information

### 2. Leverage Snakemake's Automatic Job Naming

**Decision**: Customize Snakemake worker job names using the `analysis_id` field from config.

**How Snakemake names jobs** (discovered via exploration):
- Orchestrator job: Hardcoded in sbatch script as `triton_snakemake_orchestrator`
- Worker jobs: Auto-generated by Snakemake SLURM executor based on rule names
- Default pattern: `smk-{rule}-{jobid}` or similar (Snakemake-internal)

**Customization approach**:
- Add `job-name` field to `slurm.sbatch` configuration in generated profile
- Use pattern: `{cfg.analysis_id}_{rule}` to make all worker jobs identifiable
- Example: `norfolk_multi_sim_run_tritonswmm` for analysis_id="norfolk_multi_sim"

**Implementation location**: `workflow.py:generate_snakemake_config()`, lines 605-648

```python
# Add to slurm.sbatch section
config["slurm"]["sbatch"]["job-name"] = f"{self.cfg_analysis.analysis_id}_{{rule}}"
```

**Result**: All worker jobs for an analysis will share the `analysis_id` prefix, making them queryable via:
```bash
squeue -u $(whoami) --name "norfolk_multi_sim_*"
scancel --name "norfolk_multi_sim_*"  # Cancel all workers
```

### 3. Customize Orchestrator Job Name

**Decision**: Make orchestrator job name include `analysis_id` for easy identification.

**Current state**: Hardcoded as `triton_snakemake_orchestrator` (line 1751 in workflow.py)

**New pattern**: `{analysis_id}_orchestrator`

**Example**: `norfolk_multi_sim_orchestrator`

**Benefits**:
- Instantly identify which analysis each orchestrator belongs to
- `squeue` output shows meaningful names
- Enables targeted cancellation: `scancel --name norfolk_multi_sim_orchestrator`

### 4. Cancel Method Behavior

**Method signature**:
```python
def cancel(self, verbose: bool = True, wait_timeout: int = 30) -> dict:
    """
    Cancel ongoing SLURM workflow for this analysis.

    Only works for multi_sim_run_method == "batch_job".

    Returns dict with:
        - success: bool
        - orchestrator_canceled: bool
        - workers_canceled: bool
        - jobs_were_running: bool
        - message: str
        - orchestrator_job_id: str | None
        - errors: list[str]
    """
```

**Validation**:
- Fail fast if `multi_sim_run_method != "batch_job"`
- Check if jobs are actually running before attempting cancellation
- Gracefully handle already-completed workflows (no user conditionals needed)

**Cancellation strategy**:
1. **Pre-check**: Query `squeue` to verify jobs are actually running
2. **Early exit**: If no jobs running, print informative message and return success
3. **Cancel orchestrator**: Send `scancel {orchestrator_job_id}` if orchestrator running
4. **Cancel workers**: Send `scancel --name "{analysis_id}_*"` if workers running
5. **Wait for termination**: Poll `squeue` every 2s until jobs disappear (up to `wait_timeout`)
6. **Verify completion**: Confirm all targeted jobs have terminated
7. **Update log**: Set cancellation flag and timestamp

**Key improvements**:
- **No user conditionals needed**: User can always call `cancel()` without checking status first
- **Verification**: Waits for jobs to actually terminate, not just sending signals
- **Informative messages**: Clear output when jobs aren't running vs. successful cancellation
- **Timeout handling**: Warns if jobs don't terminate within timeout but doesn't fail
- **Return data**: `jobs_were_running` flag tells user if there was anything to cancel

**Error handling**:
- Gracefully handle already-completed jobs (not an error)
- Report partial failures (e.g., orchestrator canceled but workers timed out)
- Log stderr from `scancel` commands
- Handle SLURM query timeouts

### Phase 4: Implement `analysis.cancel()` Method

**File**: `src/TRITON_SWMM_toolkit/analysis.py`

**Add new method** (around line 1700, after `submit_workflow()`):

```python
def cancel(self, verbose: bool = True, wait_timeout: int = 30) -> dict:
    """
    Cancel ongoing SLURM workflow for this analysis.

    This method cancels both the orchestrator job and all worker jobs submitted
    by Snakemake. Only works for multi_sim_run_method == "batch_job".

    The method uses persistent log data to identify jobs, so it works across
    terminal sessions (close terminal, reopen, reinitialize analysis, call cancel).

    **Key features:**
    - Checks if jobs are actually running before attempting cancellation
    - Waits for cancellation to complete and verifies jobs are terminated
    - Gracefully handles already-completed workflows
    - Returns informative messages when no jobs are running

    Parameters
    ----------
    verbose : bool, default=True
        Print progress messages
    wait_timeout : int, default=30
        Maximum seconds to wait for job cancellation verification

    Returns
    -------
    dict
        Cancellation status with keys:
        - success: bool (True if cancellation succeeded or no jobs running)
        - orchestrator_canceled: bool
        - workers_canceled: bool
        - jobs_were_running: bool (False if no jobs found to cancel)
        - message: str
        - orchestrator_job_id: str | None
        - errors: list[str] (any errors encountered)

    Raises
    ------
    ValueError
        If multi_sim_run_method is not "batch_job"
    """
    import subprocess
    import datetime
    import time
    import re

    # Validation: Only supported for batch_job mode
    if self.cfg_analysis.multi_sim_run_method != "batch_job":
        raise ValueError(
            f"cancel() only supported for multi_sim_run_method='batch_job'. "
            f"Current mode: '{self.cfg_analysis.multi_sim_run_method}'"
        )

    if verbose:
        print(f"[Cancel] Checking workflow status for analysis '{self.cfg_analysis.analysis_id}'", flush=True)

    # Load orchestrator job ID from persistent log
    orchestrator_job_id = self.log.orchestrator_job_id.get()
    analysis_id = self.cfg_analysis.analysis_id

    # Step 0: Check if jobs are actually running
    orchestrator_running = False
    workers_running = False

    if orchestrator_job_id:
        # Check orchestrator status
        result = subprocess.run(
            ["scontrol", "show", "job", orchestrator_job_id],
            capture_output=True,
            text=True,
            timeout=5,
        )

        if result.returncode == 0:
            # Job still in SLURM queue
            orchestrator_running = True
        elif verbose:
            print(f"[Cancel] Orchestrator job {orchestrator_job_id} not found in queue (already completed)", flush=True)

    # Check for active worker jobs
    result = subprocess.run(
        ["squeue", "-u", "$(whoami)", "-o", "%j", "-h"],
        capture_output=True,
        text=True,
        shell=True,
        timeout=5,
    )

    if result.returncode == 0:
        worker_count = 0
        for line in result.stdout.split("\n"):
            if analysis_id in line and "orchestrator" not in line:
                worker_count += 1

        if worker_count > 0:
            workers_running = True
            if verbose:
                print(f"[Cancel] Found {worker_count} active worker jobs", flush=True)

    # Early exit if no jobs running
    if not orchestrator_running and not workers_running:
        if verbose:
            print(f"[Cancel] No jobs currently running for analysis '{analysis_id}'", flush=True)
            if orchestrator_job_id:
                print(f"[Cancel] (Last submitted orchestrator job {orchestrator_job_id} has completed)", flush=True)

        return {
            "success": True,
            "orchestrator_canceled": False,
            "workers_canceled": False,
            "jobs_were_running": False,
            "orchestrator_job_id": orchestrator_job_id,
            "analysis_id": analysis_id,
            "message": f"No active jobs found for analysis '{analysis_id}'",
            "errors": [],
        }

    # Jobs are running, proceed with cancellation
    if verbose:
        print(f"[Cancel] Canceling workflow for analysis '{analysis_id}'", flush=True)

    errors = []
    orchestrator_canceled = False
    workers_canceled = False

    # Step 1: Cancel orchestrator job by ID
    if orchestrator_running:
        if verbose:
            print(f"[Cancel] Canceling orchestrator job {orchestrator_job_id}...", flush=True)

        result = subprocess.run(
            ["scancel", orchestrator_job_id],
            capture_output=True,
            text=True,
        )

        if result.returncode == 0:
            if verbose:
                print(f"[Cancel]   ✓ Cancellation command sent", flush=True)
        else:
            error_msg = f"Failed to send cancel to orchestrator job {orchestrator_job_id}: {result.stderr.strip()}"
            errors.append(error_msg)
            if verbose:
                print(f"[Cancel]   ✗ {error_msg}", flush=True)

    # Step 2: Cancel all worker jobs by name pattern
    if workers_running:
        worker_pattern = f"{analysis_id}_*"
        if verbose:
            print(f"[Cancel] Canceling worker jobs matching pattern '{worker_pattern}'...", flush=True)

        result = subprocess.run(
            ["scancel", "--name", worker_pattern],
            capture_output=True,
            text=True,
        )

        if result.returncode == 0:
            if verbose:
                print(f"[Cancel]   ✓ Cancellation command sent", flush=True)
        else:
            stderr = result.stderr.strip()
            if stderr and "No jobs found" not in stderr and "Invalid job" not in stderr:
                error_msg = f"Failed to send cancel to worker jobs: {stderr}"
                errors.append(error_msg)
                if verbose:
                    print(f"[Cancel]   ✗ {error_msg}", flush=True)

    # Step 3: Wait for jobs to actually terminate
    if verbose:
        print(f"[Cancel] Waiting for jobs to terminate (timeout: {wait_timeout}s)...", flush=True)

    start_time = time.time()
    check_interval = 2  # seconds between checks

    while time.time() - start_time < wait_timeout:
        time.sleep(check_interval)

        # Check if orchestrator terminated
        if orchestrator_running and not orchestrator_canceled:
            result = subprocess.run(
                ["scontrol", "show", "job", orchestrator_job_id],
                capture_output=True,
                text=True,
                timeout=5,
            )
            if result.returncode != 0:
                # Job no longer in queue
                orchestrator_canceled = True
                if verbose:
                    print(f"[Cancel]   ✓ Orchestrator job {orchestrator_job_id} terminated", flush=True)

        # Check if workers terminated
        if workers_running and not workers_canceled:
            result = subprocess.run(
                ["squeue", "-u", "$(whoami)", "-o", "%j", "-h"],
                capture_output=True,
                text=True,
                shell=True,
                timeout=5,
            )

            worker_count = 0
            if result.returncode == 0:
                for line in result.stdout.split("\n"):
                    if analysis_id in line and "orchestrator" not in line:
                        worker_count += 1

            if worker_count == 0:
                workers_canceled = True
                if verbose:
                    print(f"[Cancel]   ✓ All worker jobs terminated", flush=True)

        # Check if all done
        if (not orchestrator_running or orchestrator_canceled) and (not workers_running or workers_canceled):
            break

    # Final verification
    remaining_jobs = []

    if orchestrator_running and not orchestrator_canceled:
        remaining_jobs.append(f"orchestrator job {orchestrator_job_id}")

    if workers_running and not workers_canceled:
        # Count remaining workers
        result = subprocess.run(
            ["squeue", "-u", "$(whoami)", "-o", "%j", "-h"],
            capture_output=True,
            text=True,
            shell=True,
            timeout=5,
        )
        worker_count = 0
        if result.returncode == 0:
            for line in result.stdout.split("\n"):
                if analysis_id in line and "orchestrator" not in line:
                    worker_count += 1
        if worker_count > 0:
            remaining_jobs.append(f"{worker_count} worker jobs")

    if remaining_jobs:
        error_msg = f"Timeout: {', '.join(remaining_jobs)} did not terminate within {wait_timeout}s"
        errors.append(error_msg)
        if verbose:
            print(f"[Cancel]   ⚠ {error_msg}", flush=True)
            print(f"[Cancel]   (Jobs may still be terminating in background)", flush=True)

    # Step 4: Update analysis log
    self.log.workflow_canceled.set(True)
    self.log.workflow_cancellation_time.set(datetime.datetime.now().isoformat())

    success = (
        (not orchestrator_running or orchestrator_canceled) and
        (not workers_running or workers_canceled) and
        len(errors) == 0
    )

    if verbose:
        if success:
            print(f"[Cancel] ✓ Workflow canceled successfully", flush=True)
        else:
            print(f"[Cancel] ✗ Cancellation completed with warnings/errors", flush=True)

    return {
        "success": success,
        "orchestrator_canceled": orchestrator_canceled,
        "workers_canceled": workers_canceled,
        "jobs_were_running": True,
        "orchestrator_job_id": orchestrator_job_id,
        "analysis_id": analysis_id,
        "message": "Workflow canceled" if success else f"Cancellation issues: {'; '.join(errors)}",
        "errors": errors,
    }
```

### Phase 5: Add Convenience Method for Checking Workflow Status

**File**: `src/TRITON_SWMM_toolkit/analysis.py`

**Add helper method** (after `cancel()`):

```python
def get_slurm_job_status(self, verbose: bool = False) -> dict:
    """
    Check status of submitted SLURM workflow.

    Returns information about the orchestrator job and worker jobs.

    Parameters
    ----------
    verbose : bool, default=False
        Print detailed status information

    Returns
    -------
    dict
        Status information with keys:
        - orchestrator_job_id: str | None
        - orchestrator_status: str | None (RUNNING, PENDING, COMPLETED, etc.)
        - submission_time: str | None
        - canceled: bool
        - active_workers: int (count of running worker jobs)
    """
    import subprocess
    import re

    orchestrator_job_id = self.log.orchestrator_job_id.get()
    submission_time = self.log.orchestrator_submission_time.get()
    canceled = self.log.workflow_canceled.get() or False

    # Check orchestrator status
    orchestrator_status = None
    if orchestrator_job_id:
        try:
            result = subprocess.run(
                ["scontrol", "show", "job", orchestrator_job_id],
                capture_output=True,
                text=True,
                timeout=5,
            )

            if result.returncode == 0:
                # Parse JobState from scontrol output
                for line in result.stdout.split("\n"):
                    if "JobState=" in line:
                        match = re.search(r"JobState=(\S+)", line)
                        if match:
                            orchestrator_status = match.group(1)
                            break
            else:
                orchestrator_status = "COMPLETED"  # Not in queue
        except (subprocess.TimeoutExpired, FileNotFoundError):
            orchestrator_status = "UNKNOWN"

    # Count active worker jobs
    active_workers = 0
    analysis_id = self.cfg_analysis.analysis_id
    try:
        result = subprocess.run(
            ["squeue", "-u", "$(whoami)", "-o", "%j", "-h"],
            capture_output=True,
            text=True,
            shell=True,
            timeout=5,
        )

        if result.returncode == 0:
            for line in result.stdout.split("\n"):
                if analysis_id in line and "orchestrator" not in line:
                    active_workers += 1
    except (subprocess.TimeoutExpired, FileNotFoundError):
        pass

    status = {
        "orchestrator_job_id": orchestrator_job_id,
        "orchestrator_status": orchestrator_status,
        "submission_time": submission_time,
        "canceled": canceled,
        "active_workers": active_workers,
    }

    if verbose:
        print(f"\n{'='*60}")
        print(f"Workflow Status: {self.cfg_analysis.analysis_id}")
        print(f"{'='*60}")
        print(f"Orchestrator Job ID: {orchestrator_job_id or 'Not submitted'}")
        print(f"Orchestrator Status: {orchestrator_status or 'N/A'}")
        print(f"Submission Time:     {submission_time or 'N/A'}")
        print(f"Canceled:            {canceled}")
        print(f"Active Workers:      {active_workers}")
        print(f"{'='*60}\n")

    return status
```

## Verification Steps

1. **Submit batch job workflow**:
   ```python
   tk = Toolkit("system.yaml", "analysis.yaml")
   result = tk.analysis.submit_workflow(wait_for_completion=False)
   ```

2. **Verify job names in squeue**:
   ```bash
   squeue -u $USER
   # Should show: {analysis_id}_orchestrator, {analysis_id}_run_triton, etc.
   ```

3. **Verify persistent logging**:
   ```bash
   cat {analysis_dir}/log.json | jq '.orchestrator_job_id'
   # Should show: "9428652" (or actual job ID)
   ```

4. **Close terminal, reopen, and cancel**:
   ```python
   tk = Toolkit("system.yaml", "analysis.yaml")
   status = tk.analysis.get_slurm_job_status(verbose=True)
   tk.analysis.cancel(verbose=True)
   ```

5. **Verify jobs canceled**:
   ```bash
   squeue -u $USER
   # Should show no jobs matching analysis_id
   ```

6. **Check cancellation logged**:
   ```bash
   cat {analysis_dir}/log.json | jq '.workflow_canceled'
   # Should show: true
   ```

## Benefits

1. **Cross-session cancellation**: Works after closing/reopening terminal
2. **Simple API**: Just `analysis.cancel()` - no manual job ID tracking or status checking needed
3. **No user conditionals required**: User can always call `cancel()` without checking if jobs are running first
4. **Verified cancellation**: Waits for jobs to actually terminate, not just sends signals
5. **Clear job names**: Easy to identify which jobs belong to which analysis in `squeue`
6. **Follows existing patterns**: Uses established LogField[T] infrastructure
7. **No external dependencies**: Pure SLURM + existing toolkit infrastructure
8. **Selective cancellation**: Can cancel specific analyses without affecting others
9. **Status checking**: `get_slurm_job_status()` shows what's running (optional, not required for cancel)
10. **Robust error handling**: Graceful degradation when jobs already completed
11. **Informative output**: Clear messaging about what was canceled vs. nothing running
